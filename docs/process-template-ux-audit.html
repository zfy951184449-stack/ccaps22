  • ❌ 但源码审查暴露了 DOM 审计看不到的真实缺陷（见下方"甘特图源码审查"章节）

🔬 十、甘特图前端源码深度审查 (v2.0 新增)

10.1 技术架构概览

甘特图核心位于 frontend/src/components/wxb-ui/GanttChart/（约5500行），由 ProcessTemplateV3Editor 装配。采用 单 Canvas + RAF 渲染循环 架构：

  • 渲染: useGanttRenderer — 5层绘制管道 (网格 → 时间轴 → 依赖线 → 任务条 → 拖拽叠层)，dirty 标志条件重绘，DPR 高清适配
  • 拖拽: useGanttDrag — Canvas原生拖拽(无DOM ghost)，RAF驱动
  • 命中测试: useGanttHitTest — 坐标→任务/组，memoized 行/跨度索引
  • 状态: useGanttStore — useReducer + stateRef 镜像(供RAF闭包读取最新值)，19个action
  • 业务接线: useV3EditorActions — 拖拽/resize/删除/自动排程的API调用

整体架构评价: 设计成熟，类型完整(9/10)，性能意识良好。这是一个相当专业的自研甘特图实现。

10.2 已确认的强大功能（源码佐证）

✓ 拖拽系统（useGanttDrag.ts）

  • 单任务移动 / 多选批量移动(Ctrl/Shift多选) / 组级联移动(BFS子孙) / 双边缘 resize
  • 15分钟吸附 (SNAP_HOURS=0.25)，拖拽阈值 5px(任务)/12px(组) 防误触
  • 3级警告等级(normal/warning/danger，>1h警告、>4h危险)
  • 撤销栈 (MAX_UNDO=10)，ESC取消，Ctrl+Z撤销，级联拖拽后弹3秒撤销Toast
  • 时间窗约束 clamp(windowStart/windowEnd)，边缘自动滚动(40px触发区)
  • 异步校验回滚: onDragEnd 返回 false 时回滚——后端拒绝可优雅处理

✓ 视图与缩放（GanttToolbar + GanttCanvas + useGanttStore）

  • 4种视图模式: 时(600px)/天(120px)/周(60px)/月(40px)
  • Ctrl+滚轮缩放(40–600px，以光标为锚点)，普通滚轮平移
  • 双击时间轴进入"单日展开"模式，含上一日/下一日导航
  • collapseEmptyNightShifts 折叠空夜班时段(21:00–09:00)，节省横向空间

✓ 高级业务功能

  • 共享组: Union-Find 传递闭包计算，按天分配局部唯一颜色，hover 联动高亮(150ms去抖)
  • 设备绑定: 右键菜单内联绑定/解绑/新建设备
  • 多选购物车面板: 按组分类显示已选，一键创建共享组/批量设备绑定
  • 依赖约束线(FS/SS/FF/SF)、冲突高亮、minimap、tooltip智能避让、自动排程

10.3 源码审查发现的真实缺陷

🔴 缺陷1: "编辑操作"是死功能（高优先级）

位置: ProcessTemplateV3Editor.tsx:361-364

问题: 右键菜单/选中后的"编辑操作"最终调用 handleTaskEdit，但它只弹一个 toast：

const handleTaskEdit = useCallback((task) => {
  // full inspector panel is a future enhancement
  message.info(`编辑 ${task.label} (ID: ${task.id})`);
}, []);

影响: 用户点"编辑操作"期望打开属性编辑面板（改名称/工时/人数/设备等），结果只看到一行提示，无法编辑。这是甘特图最核心的编辑入口之一，目前是占位实现。

建议: 实现操作属性 Inspector 面板（可复用 QuickCreateOperationModal 的字段做编辑态），优先级最高。

🔴 缺陷2: 批量"解除绑定"按钮失效（高优先级）

位置: ProcessTemplateV3Editor.tsx:726

问题: 多选面板的"解除绑定"按钮调用 handleBatchUnbind([])——传入空数组。而 handleBatchUnbind 首行 if (!scheduleIds.length) return; 直接返回，按钮点了没有任何效果。

建议: 应从甘特图当前选中集合取 taskIds 传入，而非空数组。

🟡 缺陷3: 批量设备绑定依赖脆弱的 DOM 查询（中优先级）

位置: ProcessTemplateV3Editor.tsx:574-589

问题: handleBatchBind 通过 document.querySelectorAll('[data-selected-task-id]') 抓取选中项，代码注释自己也承认是 fallback("if the above approach fails")。绕过 React 状态、直接查 DOM 属性，容易因渲染时机/属性缺失而拿到空集合，导致批量绑定静默失败。

建议: 通过 selectionPanelExtraActions 的回调把选中的 taskIds 作为参数传入，移除 DOM 查询。

🔴 缺陷4: Canvas 可访问性近乎为零（合规风险）

说明: v1.0 报告"无障碍完善"指的是外围DOM按钮有标签；但甘特图主体是 Canvas，对辅助技术完全不可见：

  • ❌ 无键盘导航(Canvas无法 Tab 聚焦、无方向键移动/选择任务)
  • ❌ 屏幕阅读器读不到任何任务条/依赖/分组(无 ARIA、无 role、无替代文本)
  • ❌ 冲突状态仅靠红色表达(color-only)，不满足 WCAG 2.1 AA

建议: 若有合规要求，需提供旁路的隐藏 HTML 语义树或 ARIA live region 同步关键状态；至少实现键盘 Tab/方向键/Enter 选择与移动。

🟡 缺陷5: 大数据量性能隐患（中优先级，100+任务时显现）

风险点位置影响
依赖线每帧全量扫描，无视口裁剪 useGanttRenderer (drawDependencies) 30+依赖线 + 拖拽时，帧率可能掉到 30–40fps
getDescendantTaskIds O(n²) 遍历找子孙 useGanttDrag.ts:72 嵌套组级联拖拽时变慢，应预建 parent→children 索引
共享组 Union-Find 颜色映射未充分缓存 GanttChart.tsx (buildShareColorMap) 100+任务 + 50+共享链接时，props变更即重算

说明: 任务条渲染已做行/列裁剪(只画可见行±1)，滚动/缩放表现良好；瓶颈集中在依赖线与级联/共享路径。当前数据量(单模板几十个操作)下无感知，规模上去后需优化。

🟡 缺陷6: 冲突检测是空占位（中优先级）

位置: useV3EditorActions.ts:161-166

问题: conflictMap 永远返回空对象，注释写明 "can be enriched once validation endpoint is integrated"。意味着甘特图当前不会真正标注约束冲突，冲突高亮能力闲置。

🟡 缺陷7: 工具栏快捷操作不齐 + 测试覆盖薄弱（低-中优先级）

  • 工具栏缺"全选/清空选择""适配视图(zoom-fit)""一键展开/折叠全部"等快捷按钮（这些功能在右键菜单里有，但工具栏未暴露，发现性差）
  • 单元测试仅覆盖 ganttUtils 等少数工具函数；拖拽/撤销/约束/共享组等核心交互缺少测试
  • 键盘快捷键(ESC/Ctrl+Z/Delete)无UI提示，用户难以发现

10.4 源码审查结论

甘特图是本系统技术含金量最高的模块，自研 Canvas 引擎在拖拽、缩放、共享组、级联编辑上的完成度远超一般业务甘特图，架构与类型质量优秀。

但存在 2个高优先级功能缺口（"编辑操作"未实现、批量解绑失效），以及可访问性、性能隐患、冲突检测占位等问题。修复建议优先级：

  1. P0: 实现"编辑操作"属性面板（缺陷1）——这是核心编辑闭环的缺失
  2. P0: 修复批量解除绑定空数组 bug（缺陷2，改动极小）
  3. P1: 重构批量绑定的 DOM 查询（缺陷3）、接入真实冲突检测（缺陷6）
  4. P2: 依赖线视口裁剪 + 级联索引优化（缺陷5）、补齐工具栏与测试（缺陷7）
  5. P2/合规驱动: Canvas 可访问性方案（缺陷4）
