# V5 求解器高级配置重构——视觉设计简报

> 文档类型：交付给视觉设计师（Claude Design）的产品设计简报
> 编写角色：资深 B 端产品设计师
> 编写日期：2026-06-12
> 上游必读：`docs/solver_v5/design/29_config_current_state.md`（现状审计，58 接口字段逐字段盘点 + 9 痛点）、`docs/solver_v5/design/12_frontend_design.md` §5 设计语言（wxb-ui 组件清单 + token + 监视器视觉）、`docs/scheduling_principles.md`（业务原则）
> 冻结铁律（继承 20_IMPLEMENTATION_PLAN R2/R3）：**87 键语义/键名/默认值一字不改，所有新功能纯加法**。本次重构改的是「外壳与编排」，不动「字段契约」。

---

## 0. 一句话定位

把一个「87 字段单层长滚动、刷新即丢、无方案、无校验、无联动」的配置弹窗，重构成一个**以「配置方案（Scheme）」为中心**的可持久化、可分层、可校验、可与求解结果双向联动的配置工作台——核心用户（排班计划员）的真实诉求是「我上个月调好的那套参数，这个月还能一键拿回来」。

**设计主张**：配置不再是「打开弹窗 → 临时调一堆开关 → 关掉就忘」，而是「选一个方案 → 微调 → 一键试算 → 求解 → 结果固化快照」。方案是一等公民，字段是方案的内容。

## 1. 功能集（P0 / P1 / P2 分级）

> 每项含：用户故事（谁/什么场景）+ 为什么需要 + 验收要点。末尾列出「砍掉的功能」并说明。

### P0 —— 一期最小闭环（解决「刷新即丢」这个致命痛点）

#### P0-1 配置方案管理（命名保存 / 默认方案 / 复制 / 重命名 / 删除 / 最近使用自动恢复）

- **用户故事**：排班计划员小李每月初为下个月排班。上个月她花 20 分钟把「夜班间隔、领导策略、工时偏差权重」调到适合本厂的状态，求解满意。这个月打开配置，**期望直接看到上次那套参数**，而不是又从默认值从头调一遍。
- **为什么需要**：这是老板点名的「客户真正需要的功能」，也是 9 痛点里最致命的 P3（配置不持久，刷新即丢）。计划员是「每月高频」核心用户，重复劳动的边际成本极高。没有方案管理，前面所有 UX 优化都是给「一次性临时态」化妆。
- **验收要点**：
  1. 能把当前 58 字段配置「另存为方案」并命名（如「6 月常规排班」）。
  2. 列出我的全部方案，点击即加载，加载后 58 字段全部就位。
  3. 可设一个方案为「我的默认方案」——下次打开配置弹窗自动加载它（而非系统默认值）。
  4. 复制（生成「副本」）、重命名、删除。删除有 `WxbPopconfirm` 二次确认。
  5. **最近使用自动恢复**：即使用户从没点「保存」，离开前的工作态也要进 `localStorage` 草稿，刷新/重进后弹一个 `WxbToast`「已恢复上次未保存的配置 [放弃] [保留]」。这是兜底，防止「忘了存」的数据丢失。
  6. 持久化必须落后端（见 §4），不能只靠 localStorage——否则换电脑/换浏览器即丢。localStorage 仅作「未保存草稿」与离线兜底。

#### P0-2 内置模板（「常规月度 / PPQ 冲刺 / 节假日宽松」三套出厂预设）

- **用户故事**：新来的计划员小王第一次用，不知道「PPQ（工艺验证批）月份该把领导策略放宽到 soft、把工时容差调大」。他希望有一个下拉「从模板新建」，选「PPQ 冲刺」就得到一套已经调好的合理起点。
- **为什么需要**：CMO 工厂的排班有强季节性——常规月 vs 验证批冲刺月 vs 多法定节假日的月份（如春节/国庆），约束松紧诉求差异巨大（参考 `scheduling_principles.md`：节假日三倍薪管控、月度工时 ±8h 容差可调）。模板把「专家知识」固化下来，降低新人门槛，也给老手一个干净的复位基准。
- **验收要点**：
  1. 出厂内置 3 个只读模板：`常规月度`（= 系统默认）、`PPQ 冲刺`（领导策略放宽、最大连续工作天数放宽、工时偏差权重降低）、`节假日宽松`（允许周末补班、三倍薪权重降低、独立任务空缺容忍度提高）。
  2. 模板**只读**，不可被用户改写（改了会提示「另存为新方案」）。
  3. 模板内容用现有 58 键表达，**不新增任何键**（纯参数组合）。每个模板需视觉设计师配一句话副标题说明适用场景。
  4. 模板的具体取值由产品/系统管理员在交付前敲定（见 §6 开放问题）——视觉只需预留「模板卡片」展示位。

#### P0-3 改动徽标 + 与默认/方案的 diff 提示（「已修改 N 项」）

- **用户故事**：计划员调了一通后，想知道「我到底动了哪几个开关」，避免误改埋雷。
- **为什么需要**：痛点 P3「无变更标记，用户不知道改了哪些字段」。GMP 文化下「我改了什么」是基本可追溯诉求。这是 P0 里最轻量但价值极高的一项。
- **验收要点**：
  1. 顶栏常驻一枚 `WxbBadge`「已修改 N 项」（相对当前加载的方案 / 系统默认）。N=0 时不显示或显示「未修改」。
  2. 被修改的字段行有视觉标记（左侧一条 `var(--wx-blue-600)` 竖条 + 字段标签加「已改」`WxbTag`）。复用现有 `.solver-v5-config-row-highlight` 同源的视觉语言但区分色（高亮跳转用 amber，diff 用 blue）。
  3. 点徽标 → 弹出一个轻量「改动清单」浮层（字段名 / 默认值 → 当前值），即简版 diff。

#### P0-4 分层呈现（常用快捷面板 + 专家全量 + 锚点导航）

- **用户故事**：计划员 90% 的月份只需要碰 5~8 个字段（求解时长、工时容差、领导策略、夜班间隔、是否允许空缺）。她不想为了改一个「夜班间隔」滚过 11 个硬约束开关。
- **为什么需要**：痛点 P1（2000px 长滚动、3 屏才看完）、P6（无折叠、看不出哪些是高频）。87 字段对核心用户是噪声。**信息分层是把「专家全量」让位给「计划员高频」的关键**。
- **验收要点**：
  1. 进入配置默认落在「常用」视图：一个精选的高频字段快捷面板（约 8~10 项，含求解参数、工时容差概念、领导策略三态、夜班间隔、允许空缺、独立任务）。
  2. 一键切到「专家」视图（`WxbSegmented`：常用 / 专家），呈现全部分组卡片。
  3. 专家视图左侧有锚点导航（分组列表，点击滚动定位 + 当前分组高亮，类似目录），解决「找字段靠目测滚」。
  4. 顶部字段搜索框（`WxbSearchInput`），输入关键词（如「夜班」「领导」「权重」）即过滤/定位匹配字段。**搜索是 P0 还是 P1 见取舍说明**——本简报判定为 P0，因为它是「87 字段可用性」的最低成本解药，实现上只是对已有字段做客户端过滤，无后端依赖。

> **P0 范围的论证**：P0 = 「让计划员每月省 20 分钟」+「永不丢配置」+「降低误操作」。方案管理（P0-1）+ 模板（P0-2）+ 改动可见（P0-3）+ 分层（P0-4）构成一个能独立上线、客户立即有感的闭环。校验、diff 对比、与求解联动、审计——价值真实但属于「闭环之上的增强」，下沉到 P1。

### P1 —— 二期增强（让配置「更稳更专业」）

#### P1-1 校验与防呆（范围校验 / 冲突组合检测 / 危险开关二次确认）

- **用户故事**：计划员把「最少连续上班 3 天」「最多连续上班 2 天」填反了，或把 `allow_position_vacancy` 关了同时又关掉了某硬约束，导致必然无解。她希望保存/试算前就被拦下，而不是跑 5 分钟求解才发现无解。
- **为什么需要**：痛点 P4（无输入校验，min/max 仅 UI 层互锁不做提交校验，开关关闭后权重静默保留），痛点 P7（`enable_balance_night_shifts` 双重出现易误解）。冲突组合是 INFEASIBLE 的常见根因，提前拦截比事后诊断便宜得多。
- **验收要点**：
  1. **范围校验**：pattern 系列 min≤max、夜班间隔下限等，保存/试算时统一校验，违规字段红框 + 行内错误文案。
  2. **冲突组合检测**：维护一张「已知危险组合」规则表（如「关闭 X 又关闭 Y 极可能无解」「领导全 ban 但开启领导必在岗」），命中时顶部黄色 `WxbAlert` 警示（非阻断，可强制继续）。
  3. **危险开关二次确认**：对会显著改变结果性质的开关（如关闭 `enable_standard_hours` 工时合规、`strict_locked_shifts` 严格模式）加 `WxbPopconfirm`「此开关可能导致排班违规/无解，确认关闭？」。
  4. 解决 P7：`enable_balance_night_shifts` 在「夜班约束」与「优化目标」两处出现时，视觉上明确标注「同一开关，两处联动」，避免误认两个功能。

#### P1-2 权重可读化（原始数字 → 相对优先级可视化，底层键值语义不变）

- **用户故事**：计划员看到「空缺惩罚 10000、工时偏差 1」一头雾水，不知道把 10000 改成 20000 意味着什么，更不敢动。
- **为什么需要**：痛点 P2「权重跨 4 个数量级、无相对比例、反直觉」。这是「专家功能」里最劝退的一块。
- **验收要点**：
  1. 在权重数字框旁增加「相对优先级」可视化：把同组权重归一化为「优先级条」或「高/中/低」语义标签（如 vacancy=10000 显示为「最高优先」，deviation=1 显示为「最低」）。这是**纯展示层换算**，写回后端的仍是原始整数。
  2. 提供「高级模式」开关，专家可切回看原始数字直接编辑。计划员默认看相对优先级。
  3. **铁律**：任何可读化都不得改变写入后端的键值——`objective_weight_vacancy` 落库永远是那个整数。可读化只是 UI 投影（参考 12_frontend §3.6「前端永不直接改目标权重语义」的同源原则）。

#### P1-3 内嵌「预检试算」按钮（联动 /precheck）

- **用户故事**：计划员调完配置，想在正式 5 分钟求解前，先花几秒确认「这套配置 + 当前批次选择」不会一上来就因数据问题报错。
- **为什么需要**：现状预检（precheck）只在「操作审查弹窗」里触发，配置弹窗内无入口。把试算搬进配置上下文，形成「调参 → 立即验」的快速回路，避免「配置-关闭-审查-发现错-重开配置」的长链路。
- **验收要点**：
  1. 配置底部操作条增加「预检试算」按钮，调 `/api/v5/scheduling/precheck`，把当前 config + 当前批次范围传入。
  2. 结果以 `WxbAlert` 条（pass/warn/error，复用现有预检告警栏视觉，见 12_frontend §7.2）就地展示，error 项关联到具体字段。
  3. 预检不落库、不开求解，纯校验旁路。

#### P1-4 与求解结果联动（run 配置快照 / 复用此次 run 配置 / 无解诊断跳入高亮融入新布局）

- **用户故事**：(a) 主管回看上周某次满意的求解，想知道「当时用的什么配置」并「照这套再跑一次本月」。(b) 计划员遇到无解，监视器诊断面板说「放宽领导策略」，点「跳到配置」，期望直接定位高亮到 `leader_ops_policy_*`。
- **为什么需要**：痛点「与求解结果无联动」。现状已有无解诊断高亮跳转（`highlightKeys` + `.solver-v5-config-row-highlight`），但 (a) run 配置快照回看 / (b)「复用此次 run 配置」尚无。GMP 下「这版排班是用哪套参数跑出来的」是强追溯诉求。
- **验收要点**：
  1. 每个 run 固化其完整 config 快照（后端已传 request_json，见 §4），结果页/历史可「查看本次配置」（只读 diff 视图）。
  2. 「复用此次 run 的配置」按钮：把该 run 的 config 加载进配置工作台（可另存为方案）。
  3. **无解诊断跳入高亮**在新布局中的落点：诊断面板点「跳到配置」→ 若目标字段在「常用」面板则就地高亮；若在「专家」分组，自动切到专家视图 + 锚点滚动 + 高亮（amber）。保留现有 `highlightKeys` 机制，只是适配分层布局。

#### P1-5 方案作用域（个人 / 团队共享 / 系统默认）

- **用户故事**：主管把自己调好的「夜班政策」方案设为「团队共享」，组内 3 个计划员都能用同一套基准；系统管理员维护「系统默认模板」。
- **为什么需要**：用户上下文明确有三类角色（计划员 / 主管 / 管理员）。共享避免「每人各调一套、口径不一」。
- **验收要点**：
  1. 方案有 `scope` 字段：`personal`（仅自己可见可改）/ `team`（团队可见，需权限改）/ `system`（系统默认/内置模板，仅管理员可改）。
  2. **权限与现有 RBAC 对齐**（重要约束）：当前 RBAC 已落库，APS 域有 `APS_SCENARIO_READ/WRITE/APPROVE` 三个权限码。**建议复用 APS_SCENARIO 权限族**，不新增权限码（与 12_frontend §1.6「V5 不新增权限码避免动 RBAC」一致）：读方案=READ、写团队/系统方案=WRITE、（可选）发布团队方案=APPROVE。个人方案不需特殊权限（owner 即可）。
  3. 视觉上方案列表分「个人 / 团队共享 / 系统默认」三段，团队/系统方案对无写权限者显示只读锁标。
- **取舍**：作用域放 P1 而非 P0，因为它依赖 RBAC 联调 + 后端 owner/scope 鉴权，工程不确定性高于「个人方案持久化」。P0 先把「个人方案」跑通（scope 字段预留但仅个人态），P1 再开团队/系统。

### P2 —— 三期（专业/合规增强，价值真实但非闭环必需）

#### P2-1 diff 视图（当前 vs 默认 / 方案 A vs 方案 B）

- **用户故事**：主管想搞清「PPQ 冲刺模板」和「常规月度」到底差哪几项；或把当前临时态和某方案对比。
- **为什么需要**：P0-3 已给「相对默认的改动徽标」轻量 diff；P2 给完整双栏对比。专业但低频。
- **验收要点**：选两个对象（方案/默认/当前态）并排，仅列出有差异的字段，左右值高亮，支持「把 A 的某项应用到当前」。

#### P2-2 导入 / 导出 JSON

- **用户故事**：管理员要把生产环境调好的一套方案，搬到另一台单机部署（本项目是单机锁定 MacBook 部署）。
- **为什么需要**：跨环境迁移、备份、给厂商发支持工单时附配置。低频但对运维有用。
- **验收要点**：导出当前方案为 `.json`（含 58 键 + 方案元数据）；导入时**做键名/版本校验**（防止导入改了键名的非法 JSON 污染契约），非法键忽略并提示。
- **取舍**：放 P2 因为单机内置方案管理已覆盖 90% 场景，JSON 导入导出主要服务跨机迁移这个长尾。

#### P2-3 变更历史 / 审计（方案的每次保存留痕）

- **用户故事**：质量审计时被问「这套排班参数 3 月到 6 月改过几次、谁改的、改了啥」。
- **为什么需要 + GMP 论证**：本项目是**生物制药 CMO（GMP 环境）**，配置变更可追溯是合规文化的一部分。**但是否 P0？判定为 P2，理由**：(1) 当前系统**尚无 auth/RBAC 中间件实际接管**（CLAUDE.md 明确「无 auth/authz/RBAC middleware wired up today」，鉴权体系仍在实施中），没有可信「操作人」身份，审计留痕缺少法律意义上的主体——**先有可信身份，审计才有价值**。(2) 真正受 GMP 监管的是「最终应用到生产的排班结果（scheduling run / scenario 的 APPROVE/PUBLISH）」，那一层已有 `aps_scenarios` 的 status 流转（DRAFT→CHECKED→APPROVED→PUBLISHED）和 approved_by 字段承载审计；**配置方案本身是「求解输入草稿」，不是受控产出**，其合规优先级低于结果审计。(3) run 配置快照（P1-4）已经为「这版排班用了什么参数」提供了事后可查的最强证据。
  - **结论**：方案级变更历史延后到 P2，待 RBAC 落地后补「保存留痕（who/when/diff）」。P0/P1 通过 run 快照 + scenario 审计已覆盖核心合规诉求。需老板确认此判断（见 §6 开放问题）。
- **验收要点**：方案详情页一条时间线（`WxbTimeline`），每次保存记 who/when + 改动摘要。

### 砍掉 / 明确不做的功能（含理由）

| 砍掉项 | 理由 |
|---|---|
| **暴露 5 个静默字段到 UI**（`max_consecutive_rest_days`、`min_rest_after_night_block`、`off_hours_multiplier`、`enable_special_shift_coverage`、`enable_special_shift_joint_coverage`，痛点 P5）| 它们沿袭 V4、求解器用硬编码默认值，**99% 用户永不需要调**。全量暴露反而加重 87 字段噪声。**折中**：在「专家」视图最底部放一个折叠的「高级/不常用参数」抽屉收纳它们，默认收起。不为它们单独做面板。键名/默认值仍冻结。 |
| **求解时间预测 / 配置对结果影响的预估** | 需要 ML 或历史数据建模，工程量大且不准，容易误导。不做。预检（P1-3）已给「能不能跑」的确定性回答。 |
| **可视化拖拽编排约束优先级（把权重做成可拖排序的列表）** | 炫但脆。权重是连续数值不是离散排序，强行拖拽会丢失精度语义，且与「键值不变」铁律冲突风险高。用 P1-2 的「相对优先级展示 + 数字直填」更稳。 |
| **多方案 A/B 求解对比（同时跑两套配置比结果）** | 属于「求解编排」范畴，不属于「配置视图」，应在求解/scenario（WHAT_IF）层做。本次重构边界是配置工作台，不越界。 |
| **配置项国际化 / 多语言** | 项目明确中文界面（CMO 工厂），不做 i18n 框架。 |
| **配置弹窗内直接改批次范围以外的数据** | 配置只管 58 键 + team_ids，不扩权到批次/人员数据。维持单一职责。 |

## 2. 信息架构与布局

### 2.1 容器选型：弹窗 vs 全屏抽屉 —— 结论「右侧全屏抽屉（`WxbDrawer` 100vw 或宽抽屉 ~960px）」

**论证**：

| 维度 | 现状 Modal 640px | 推荐 WxbDrawer |
|---|---|---|
| 内容体量 | 58 字段 + 新增方案侧栏 + diff，640px 装不下，68vh 已要滚 3 屏 | 抽屉宽 + 高（近全高），容得下「左方案栏 + 主区分组 + 右锚点」三栏 |
| 与 V5 视觉一致性 | — | **V5 监视器（SolveMonitorV5Drawer）已是 100vw 抽屉**，结果页也是全屏抽屉。配置用抽屉与 V5 体系语言统一（12_frontend §2） |
| 多区域并置（方案列表 + 字段 + 锚点） | Modal 难做三栏 | 抽屉天然适合仪表盘式三栏 |
| 打断感 | 居中 Modal 强打断 | 右滑抽屉更「侧边工作台」感，符合「调参是个过程」 |

**但保留一个折中**：「常用」视图内容很少（8~10 字段），可以用**较窄抽屉（~720px）**；切到「专家」视图时抽屉**自动加宽到 ~960px / 全屏**以容纳三栏。即「抽屉宽度随视图层级自适应」。视觉设计师需出两种宽度态。

> 不选 Modal 的根本原因：方案管理是一等公民，需要常驻的方案侧栏；Modal 的居中小窗范式无法承载「侧栏 + 主区 + 导航」的工作台结构。

## 3. 关键交互流（逐步）

### 3.a 保存为新方案

1. 用户在工作台调好若干字段，顶栏徽标显示「已修改 N 项」。
2. 点底部「另存为...」→ 弹出小 `WxbModal`：方案名（`WxbInput`，必填，查重）、作用域（`WxbRadio`：个人 / 团队共享[需权限] / —）、可选「设为我的默认方案」`WxbCheckbox`。
3. 提交 → 调 `POST /api/v5/solver-schemes`（§4）→ 成功 `WxbToast`「方案『6月常规排班』已保存」→ 顶栏方案选择器自动切到新方案，徽标归零（N=0）。
4. 若名称重复 → 行内错误「已有同名方案」，不关弹窗。

### 3.b 切换方案时存在未保存改动

1. 用户在方案 A 下改了 3 项（徽标「已修改 3 项」），从方案选择器选了方案 B。
2. 触发拦截 `WxbModal` 确认：「当前方案有 3 项未保存改动，切换将丢失。」三个动作：
   - 「保存到方案 A 再切换」（写回 A，再加载 B）
   - 「放弃改动并切换」（直接加载 B）
   - 「取消」（留在 A，不切）
3. 选择后执行。**全程不静默丢数据**——这是对痛点 P3/P8「保存语义误导」的正面回应。

### 3.c 从无解诊断跳入高亮

1. 求解返回 INFEASIBLE，监视器/结果页的 `InfeasibilityPanel` 列出冲突组，每组有「跳到配置 →」（携带 `config_keys[]`，机制沿用现状 `highlightKeys`）。
2. 点击 → 关闭监视器抽屉 → 打开配置抽屉，**携带 highlightKeys**。
3. 布局适配：
   - 若目标字段在常用面板可见 → 直接滚动到 + amber 高亮（`.solver-v5-config-row-highlight`）。
   - 若在专家分组 → **自动切到专家视图** + 展开对应折叠卡片 + 锚点滚动到位 + amber 高亮（3 秒后渐隐）。
4. 顶部加一条 `WxbAlert`（amber）「以下高亮项与本次无解相关，放宽其一可能有解」，附「全部清除高亮」。

### 3.d 配置内预检试算（P1-3）

1. 用户点底部「预检试算」。
2. 按钮进 loading，调 `/api/v5/scheduling/precheck`（传当前 config + 当前批次范围）。
3. 返回后在底部操作条上方插入一条结果区（`WxbAlert` 列表）：
   - 全 pass → 绿色「预检通过，可以求解」。
   - warn → 黄色逐条（可继续）。
   - error → 红色逐条，每条若关联字段则带「定位 →」跳到该字段高亮。
4. 预检不开求解、不落库。再次调用覆盖上次结果。

### 3.e 恢复默认（全局 vs 单分组）

1. 底部「恢复默认」是一个**带下拉的分裂按钮**（`WxbDropdown`）：
   - 主动作「恢复全部默认」→ `WxbPopconfirm`「将把全部 58 项重置为系统默认，已修改的 N 项将丢失，继续？」→ 确认后加载 `DEFAULT_SOLVER_CONFIG_V5`，徽标归零。
   - 下拉项「仅恢复本分组默认」→ 仅把当前聚焦/展开的分组字段重置（如只重置「夜班约束」），其他保留。下拉里列出各分组名供选。
2. 恢复后被重置的字段短暂高亮（区别于 diff 蓝，用中性灰闪）提示「这些被还原了」。

> 注：现状已有「恢复默认」（全局），本流是在其上增加「单分组粒度」+ 二次确认，回应痛点 P8（避免误重置全部）。

## 4. 数据与接口草图（给实现定锚，不写代码）

> **铁律重申**：58/87 键语义冻结。本节所有新表/新端点都是**外层包装**，`config_json` 内部就是现有 `SolverConfig` 对象，一字不改。

### 4.1 方案实体（建议表名 `solver_config_schemes`，新增表）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT PK | |
| `scheme_code` | VARCHAR | 稳定标识（内置模板用固定 code 如 `BUILTIN_NORMAL`/`BUILTIN_PPQ`/`BUILTIN_HOLIDAY`） |
| `scheme_name` | VARCHAR | 显示名 |
| `scope` | ENUM(`personal`,`team`,`system`) | 作用域（P0 仅落 personal；team/system 由 P1 开） |
| `team_id` | BIGINT NULL | scope=team 时归属团队 |
| `solver_version` | VARCHAR | `v5`（隔离 V4/V5，避免跨代污染） |
| `config_json` | JSON | **整份 `SolverConfig`（58 键），契约冻结** |
| `is_builtin` | TINYINT | 内置模板=1（只读，不可删改） |
| `is_default_for_owner` | TINYINT | 该 owner 的默认方案标记 |
| `created_by` / `updated_by` | INT NULL（users.id） | RBAC 落地后填真实身份 |
| `created_at` / `updated_at` | DATETIME | |
| `(P2) version` / 关联 `solver_config_scheme_history` | — | P2 审计留痕用，P0 不建 |

> **复用既有资产的判断**：`aps_scenarios` 表（scenario_type/status/approved_by）是为「排班场景/结果」设计的，承载的是**求解产出**的审批流，**不适合**直接塞配置草稿（会混淆「输入」与「产出」语义）。建议**新建轻量 `solver_config_schemes`**，与 scenario 解耦。两者关系：一个 scenario 的某次 run 引用了某个 scheme 的快照（见 4.3）。**需后端架构确认**（见 §6）。

### 4.2 所需后端端点清单（均挂 `/api/v5/*`，相对路径）

| 端点 | 方法 | 用途 | 期数 |
|---|---|---|---|
| `/api/v5/solver-schemes` | GET | 列出当前用户可见方案（个人+团队+系统，含内置模板） | P0 |
| `/api/v5/solver-schemes` | POST | 新建方案（body: name/scope/config_json/is_default） | P0 |
| `/api/v5/solver-schemes/:id` | PUT | 更新方案（改名/改内容/改默认标记） | P0 |
| `/api/v5/solver-schemes/:id` | DELETE | 删除方案（内置/无权限拒绝） | P0 |
| `/api/v5/solver-schemes/:id/duplicate` | POST | 复制为副本 | P0 |
| `/api/v5/solver-schemes/builtin` | GET | （或并入 list）返回 3 个内置模板 | P0 |
| `/api/v5/solver-schemes/import` `/export` | POST/GET | JSON 导入导出 + 键名校验 | P2 |
| `/api/v5/scheduling/precheck` | POST | **已存在**，配置内预检复用 | P1 |
| `/api/v5/scheduling/runs/:id` （含 config 快照） | GET | 读 run 的配置快照（见 4.3） | P1 |

> 权限映射：复用 `APS_SCENARIO_READ/WRITE/APPROVE`（§1 P1-5）。个人方案不强制权限（owner 即可），团队/系统方案写操作要 WRITE。RBAC 中间件落地前，端点先放行（与现状「无 auth wired」一致），但端点设计预留 `created_by`/`scope` 鉴权钩子。

### 4.3 与 scheduling_runs 配置快照的关系

- **现状**：求解请求会写 `logs/request_{id}.json`（见 12_frontend §11.4），且 V2 期 `scheduling_runs` 有 `options_json` JSON 列。V4/V5 求解时 config 整对象下发给 solver。
- **目标关系**：
  1. 每次 solve，把**当时生效的完整 `config_json` 快照**持久化到该 run（建议 `scheduling_runs` 增列 `solver_config_snapshot JSON`，或复用 `options_json`）。这是「这版排班用了哪套参数」的权威来源，也是 P1-4「复用此次 run 配置」「查看本次配置」的数据源。
  2. 快照是**值拷贝**，不引用 scheme（方案后续改了不影响历史 run）。可选记 `source_scheme_id`（仅作血缘提示）。
  3. run 快照 + scenario 审批流（`aps_scenarios.status`/`approved_by`）共同构成 GMP 审计证据链——**这正是把「方案级审计」降到 P2 的底气**（§1 P2-3）。
- **加法保证**：新增列/新表，不改 `scheduling_runs` 既有列语义；config 内容仍是冻结的 58 键。

## 6. 分期建议（P0 一期最小可上线闭环）

**P0 一期目标：终结「刷新即丢」，让计划员的配置可命名持久、可复用、改动可见、87 字段可分层导航。**
