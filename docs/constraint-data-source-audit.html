① LeadershipCoverage · Rule 1（硬）

要求每个有生产操作的日期至少 1 名 leader 上班。代码：leadership_coverage.py:147。本次 7 月有生产操作的日期共 30 天，而团队里只有 1 名 leader（446 王如月 / TEAM_LEADER） → 等价于要求她覆盖全部 30 个生产日。

② StandardHours（硬）

7 月 23 个工作日，标准工时 184h，默认上浮 +32h → 单人月度硬上限 216h。代码：standard_hours.py:97。

最低工作班次 8h　→　单 leader 覆盖 30 个生产日 ≥ 30 × 8 = 240h
硬上限 = 216h　→　240 > 216　→　无可行解

#1　org_role 废弃字段　✓ 已修复

领导约束整条数据链读取的是 employees.org_role，但该字段是未维护的旧字段（全表默认 FRONTLINE）；真实角色在 employees.primary_role_id → employee_roles.role_code。

后果：solver 看到的人全是 FRONTLINE → if not leader_emp_ids: return 0 → 领导约束永远跳过；且 15 名管理岗 can_schedule=1，被当一线照常分配生产工序。

修复：DataAssemblerV4.ts 的 fetchEmployees 两条查询路径均 JOIN employee_roles，用 COALESCE(er.role_code, e.org_role, 'FRONTLINE') 取权威角色。solver 侧零改动（契约值域一致）。已重新构建并重启 backend。

#2　Rule 1 × StandardHours 硬冲突　✓ 已修复

Rule 1（生产日 leader 覆盖）此前是不可关闭的硬约束。当 leader 数量远少于生产日数时，必然与 StandardHours 月度工时上限冲突而 INFEASIBLE（即 run_id=1306）。

修复：把 Rule 1 改为可独立开关，默认开启（保持现状），需要时可手动关闭以获得可行解。

# leadership_coverage.py — Rule 1 现在被 config 开关包裹
if config.get("enable_leader_production_coverage", True):
    # ... 每个生产日 sum(leader_work_vars) >= 1
else:
    self.log("Rule 1: production-day leader coverage DISABLED by config.")
  • 后端/求解器：enable_leader_production_coverage（默认 True），透传无需改动。
  • 前端：求解器配置弹窗「领导层排班约束」下新增缩进子开关 「└ 生产日需领导在岗」，依赖总开关。
解决 run_id=1306：在求解器配置里关闭「生产日需领导在岗」，即可绕开与月度工时上限的冲突，重新求解可得可行解。（治本方案：增加该团队的 leader 数量，或调整 StandardHours 上浮上限。）

#3　consecutive_rest_days 契约缺口　✓ 已修复

约束 consecutive_days.py:215 的 _apply_rest_boundary 读取 hist.consecutive_rest_days，但契约 HistoricalShift 只有 consecutive_work_days，DataAssemblerV4.ts 的 fetchHistoricalShifts 也从不计算休息天数。

# consecutive_days.py — hasattr 恒为 False → 永远 return 0
for hist in data.historical_shifts:
    if hasattr(hist, 'consecutive_rest_days') and hist.consecutive_rest_days > 0:
        history_map[hist.employee_id] = hist.consecutive_rest_days
if not history_map:
    return 0   # <-- 总是走到这里

后果：「最大连续休息天数」约束在求解窗口内部的滑窗正常，但感知不到员工进入窗口前已连续休息几天——跨月 / 跨冻结区间的边界处可能放任连续休息超限。对比 work limit 有对应的 consecutive_work_days，rest 这侧是不对称的实现遗漏，且静默失效不报错。

修复：HistoricalShift 契约（及后端 V4HistoricalShift）补 consecutive_rest_days 字段；fetchHistoricalShifts 对称增算连续休息天数，并把历史回看窗口从硬编码 6 天改为自适应 max(连续工作上限, 连续休息上限, 6)+1（顺带修好 work 方向的回看不足隐患）。约束侧无需改动（已在读取）。已重启 solver + backend。
