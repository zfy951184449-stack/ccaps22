一句话判断

合理，但不是完整不可变快照。当前设计适合生产排程：模板复用主数据，批次固化执行计划，求解器消费当次 JSON。若要审计级复盘，需要额外持久化完整求解输入和主数据版本。

模板拿到什么
  • operation_id
  • 阶段、顺序、相对日期
  • 推荐时间和时间窗口
批次固化什么
  • 实际开始/结束时间
  • 窗口、持续时间、人数
  • 约束和共享组
求解器收到什么
  • operation_demands
  • 人员、资质、不可用
  • 班次、日历、资源
结果写回什么
  • scheduling_runs 结果摘要
  • batch_personnel_assignments
  • employee_shift_plans

结果应用后的阶段

1. 结果草稿
  • scheduling_runs.result_summary
  • summary_json
  • 仍是求解输出，不等于生产排班已生效
2. Apply 写入生产表
  • batch_personnel_assignments：操作到人员
  • employee_shift_plans：员工到日期/班次
  • 写入 scheduling_run_id 追溯来源
3. 后续页面读取
  • 批次甘特图、日历、人员看板
  • 锁定/历史班次会影响下一次求解
  • 删除结果不自动撤销已应用生产表

四个最需要注意的点

不是全字段快照

批次保存了计划时间和人数，但展示或求解时仍可能 join 当前 operations 取名称。

资源规则受开关影响

启用批次资源快照时优先读快照；否则可能回退到操作/模板资源规则。

求解输入未完整归档

求解结果有保存，但当次完整 SolverRequest 没看到独立留档。

删除结果不等于撤销

结果 Apply 后已经写入生产表，后续撤销需要处理人员分配和班次计划。
