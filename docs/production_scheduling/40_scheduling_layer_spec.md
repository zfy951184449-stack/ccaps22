# 批次 / 调度层规格(Scheduling Layer Spec)v0.1

> 状态:**草案 / 持续迭代**。排产系统"批次/调度层"——把工艺流模型(`10_process_flow_model_spec.md`)变成**真实可执行的排程**。
> 起草 2026-06-13。这是 §2 分层架构里的"problem + 解"那一半(模板层是 domain)。

## 1. 范围与定位

工艺流模型层(10_spec)只声明"领域词汇"(操作=需求+产出、状态机、生成规则),无时间无实例。**本层把它实例化成真实排程**:给定一个批次 + 锚点,推导/装配出操作集,在真实设备×日历上落成可行排程,撞了消解,产出喂排班。

**总基调:人机协作。** 系统**自动出一版基线**,计划员在**甘特上微调**(D-交互)。

## 2. 单批端到端流程

```
锚点(Day0, 计划员定)
  → ① 装配/派生(跑 10_spec §4 引擎)→ 固定操作集(主链 + 派生辅助)
  → ② 时序(STN):算每操作可行窗口 [最早,最晚];冲突=负权回路
       ⇅ 双向迭代
  → ③ 资源(传播 + 确定性落点, v1 无求解器):窗口内按真实设备/CIP 站/配液罐/房间/人 落点
  → ④ 消解(撞车时):换台 → 窗内重排 → 报增援(Day0 不滑)
  → ⑤ 输出:写回操作时间线 → 喂排班(solver_v4/v5)+ 冲突/增援报告
```

**关键认识(已厘清):** "争抢共享资源(CIP/配液罐)"**从单批就有**——一批 DSP 派生近百道配液 + 大量 CIP,挤有限的 CIP 站/配液罐(真实数据 Day5=16 清洗很可能就是单批高峰)。所以 ③④ 单批就要用,不是多批才有。

## 3. 锚定(DSP 为例)

- **主链各工艺步(AC Cycle/VIN/CEX/AEX/HA/UFDF/…/Bulk Fill)= 钉子**。非纯零等待(个别步有 hold window),**但钉子位置可被确定/固定**(零等待处被级联逼定;有 hold 处由工艺/计划员定位)→ 派生辅助前主链时刻已知固定。
- **只正排**:计划员给每批一个 **Day0**(由 **TAT 规定固定**),系统一律往后推。**系统不优化锚点、不倒排。**
- **交期 = 检查线**:正排结果晚于交期 → 告警(不拿交期当锚点倒排)。
- **自由度只在辅助层**:CIP/配液/房间放行 在各自窗口(最早可做 ~ 必须做完)内可挪——这才是要"安排"的。

## 4. 设备分配与连续占用

- **设备实例选择**:系统**自动选**(避争用/少换产/就近)+ **人可改**。
- **主设备跨批基本不冲突**(印证业务:"工艺设备冲突基本没有")→ 自动分配空闲实例;罕见撞了才标 + 提示(加设备/挪 Day0)。
- **连续占用(在位物料,关键约束)**:物料令牌"驻留"在某设备实例(培养液在反应器 X)→ **凡操作该令牌的工序自动绑同一台**,实例**驻留期独占、起点选一次、不可中途换**(接完种次日取样不能换台)。**判定:由"物料在位"自动推**,不用人标。
- **批次节拍**:Day0 由 TAT 规定固定;**自动排满产能 = 后续 roadmap**。

## 5. 共享资源争用与消解

- **争用对象(v0.2→修正)**:核心是 **CIP 站**;配液罐基本不紧(配制期短占 + 转储释放、数量充足),溶液效期内占的是**储存容器**。**真正"有时不够"的具体资源 = 待业务点定(§9 开放项)。** 另有 房间、WFI、人。
- **CIP 站路由(业务澄清)**:CIP 站独立、跨部门共用、**容量 1**;拓扑 **设备/罐 → 管线 → {主站, 备站}**。**引擎只往主站排;主站塞不下 → 报增援,动备站交给人**(非紧急用主站是硬规矩,备站留作干净的应急余量,排产不默认占用)。"下游 vs 培养抢 CIP" = 两条管线共用同一主站时抢该站时间轴。
- **CIP 优先级(业务澄清 C16)**:**主工艺触发的 CIP > 配液触发的 CIP**(主工艺 CIP 在钉子间、窗紧、临界;配液 CIP 效期内更宽松 → **配液让路**)。
- **消解次序(自动,CIP 池)**:① 配液 CIP **窗内调** → ② 仍冲突:配液 CIP 换站/挪 + **有限回溯**(§6.2)→ ③ **末招:主工艺 CIP 在自身 DHT/CHT 窗内微调(小影响,不动工艺钉子 / Day0)** → ④ 仍不行 → **报增援**。**Day0 永不滑;工艺钉子不动**;主设备(驻留中)不可换。CIP 末招微调是**辅助层内部让步,非反推主链**(D21 精度)。
- **增援产出**:**冲突节点 + 可执行建议**(哪个资源 / 何时 / 缺多少 / 是否动备站),**决策权交人**;人工干预入口始终保留。
- **跨批"谁先用"**:v1 **透明确定性规则**(LST 最紧者优先;计划员可标"此批优先"抬权重)+ 报增援,**不做全局求解器**;取舍 = 非全局最优,但可解释/可复算,塞不下即报人决。

## 6. 落点:目标 + 纯传播实现

### 6.1 落点目标(派生节点摆放准则,用户定 2026-06-14)

三条准则构成落点的目标函数:

1. **主链是墙,派生只填缝、永不反推主链(不变量)**:主链先排定并冻结;派生层严格下游,只占主链留下的空隙、无权重开主链。某派生必需(如"先洁净"的 CIP)塞不下 → **报增援**,引擎绝不自行挪主链那颗钉(详见 §5、`10_spec §3.6` / D21)。
2. **派生节点双侧缓冲最大化**:每个派生节点向左、向右都尽量留余量(= 时序解耦 + 启发式柔性分配,主链固定时**多项式可算**;**注:可证明最优的柔性分配是 MIP[Meng 2015],我们不做、取启发式「够稳非最优」**)。
3. **解耦遏制涟漪**:动任一节点,受影响节点数最小,且影响**被自身缓冲吃掉、不串到别人**(= 时序解耦 temporal decoupling:给节点切互不牵连的私有窗,窗内动则波及 0 节点)。

- **2 与 3 的张力(如实记)**:解耦要"花掉"共享总余量买独立性 → 全解耦 = 私有缓冲偏小,不解耦 = 窗大但一动就串。**余量充足两者兼得;余量紧张(如 Day5=16 密集)不可兼得 → 正是该报增援的点**(不挤出一碰就崩的脆计划)。全局旋钮 **解耦优先 / 肥窗优先**,默认**偏解耦 = 稳**(§9)。
- **度量**:不取"总位移最小 / 最坏位移最小",取**涟漪遏制(解耦)**——A/B/C 之争由此收口。
- **安全系数 = 涌现量**:人只给 ① contingent 边 + 范围 ② 效期硬墙 ③ 解耦旋钮;甜品区由 解耦 + **启发式 slack 分配** 涌现得出(**够稳、非可证最优**——可证最优是 MIP,不做),可解释("这步摆这儿 = 让扰动死在本地、不贴效期、不贴主链钉")。"过高安全系数反致过期"天然规避。
- **时序引擎分阶段(STNU 路线裁决,详见 `10_spec §11` / D23)**:**v1 = STN + 时序解耦**(增量 Bellman-Ford / deltastn,毫秒级)+ **标称时长**;contingent 边 day-1 进 schema 但**不跑 DC**。**v2 叠加 STNU + DC**(把"启发式够稳"升级为"可证明动态可控",可后台异步、用 CSTNU-Tool JAR 对拍)。理由:无 Py 生产级 DC 库 / 增量 DC soundness 缺陷 / ~300 操作 DC 性能墙 → v1 走 STN 正好绕开。
- **边界**:以上对**时间维度**多项式成立、纯传播;**离散资源排序**(同容量 1 CIP 站谁先谁后)仍组合难题 → v1 确定性近似,未来可选后手才在此有边际价值。

### 6.2 怎么落(纯传播实现,v1 不做求解器)

**[决定 C9 / 修订 C6·D19 — 2026-06-14]** 技术选型联网调研(四路独立 + 综合)后拍板:**v1 不上求解器,整条排产走传播模式。**

- **资源传播(time-table 扫描线)**:对每个共享池(CIP 站/配液罐/房间)按时间段累加已承诺容量,超容 → 收紧相关辅助操作的 [最早,最晚] 窗口并回写 STN(②③ 双向迭代)。绝大多数争用传播即可收紧到唯一可行。
- **确定性窗内落点**:传播后仍要在等价候选间选(哪台 skid/哪个罐/谁先)→ **透明确定性规则**:默认 **LST(最晚开始)最紧者优先**;计划员标"此批优先"则抬权重。**非全局最优,但可复算、规则透明、GMP 可辩护。**
- **按优先级增量插入(确定性、终止;C16)**:按固定优先级顺序(**主工艺 CIP 先、配液 CIP 后**)逐个插入,每步落点由前序已定者**逼定**,最后一个被完全约束 → 无自由度可选 → **天然终止、不震荡**(固定优先级打破"谁最紧"的来回拉扯 → 解审计 placement thrashing 隐患 §5.1#3)。冲突走 §5 消解阶梯。
- **报增援前先来一层廉价有限回溯(bounded swap,审计补)**:单遍 LST 贪心在**中密度争用**就可能漏解误报(非仅极密尖峰)——这是"砍求解器 + 主链墙 + Day0 不滑"三重锁死下余量零和的诚实代价。故 v1 落点 = LST 贪心 → **失败时做有限步、可解释的相邻交换/换台回溯**(仍多项式、非搜索树)→ 仍塞不下才报增援。**误报判据**(供压测):报增援窗内资源利用率 < 100% 即疑似误报,进样本。
- **报增援**:确定性落点仍塞不下 → 输出冲突节点 + 可执行建议(哪个资源/何时/缺多少),决策权交人。**Day0 永不滑。**
- **为什么砍求解器**:① 排产主体是传播问题——STN 最短路 + 资源 time-table 已能定死绝大多数窗口,求解器只在"极密争用、传播定不下排序"时有边际价值;② **可解释性(诚实收窄,审计)**:**时间窗因果维**传播更可解释(STN 最短路给"这步为何最早只能在 X"的因果链;CP-SAT 内部 SAT 子句是黑盒)——这维传播占优;但**离散排序维**("CIP A 为何排 B 前")是组合难题/确定性近似,传播**并不比求解器更可解释**,只是 **LST 规则可复算、可辩护**(求解器则能给最优性证书)——**两者都非"全局可解释最优"**;③ 无现成框架同时满足"传播可独立调用 + 拖拽毫秒级增量 + STN 可解释",传播主干须自建,既自建就不必再背求解器工程量。
- **求解器 = 未来可选逃生口(非 v1)**:若真实压测证明某些极密尖峰(如 Day5=16)确定性落点反复误报增援、而实际存在可行解,再评估把"争用子问题"打包给 OR-Tools CP-SAT 当后手(架构已留接口:那一小段窗口 + 候选已是独立子图)。**v1 范围内不实现。**
- **复用边界(D1)**:即便将来加后手,也只复用 solver_v4 的**架构模式,非代码**;模块排产专属。

## 7. 多批 / campaign

- **共享世界状态**:设备/CIP 站/配液罐跨批复用,随时间演进;某批初始态 = 锚点处对此前所有已排操作 effects 的投影快照。

- **② 主设备跨批占用 + 节拍**:基本不冲突;Day0 由 TAT 定;自动产能优化 = roadmap。
- **③ 跨批攒料(已谈,见 §10 攒批 + 物料闸)**:按工艺需求声明 scope —— **(A) 一次配制分装、单次使用、限本批**(=批内分装)/ **(B) 一次配制分装、单次使用、跨多批**(=攒批 campaign:大令牌分装服务多批领用,如 2000L 碱液 → 4×500L 给 4 批)。**同一套机制、scope 是物料/需求上的属性**;酸碱清洗剂 + 长效缓冲都套用。攒批由"物料需求单闸"承载(§10):引擎贪心提案、人审改、确认冻结。

## 10. 分阶段派发(物料需求单闸 → 任务派发闸)

排产**不是一次算到底**,而是经**两道人审闸**提交(C15):

**闸 1 · 物料需求单(料的闸)**
- 输入:装配派生结果(操作集 + 攒批提案)+ 锚点粗时间线。
- 引擎出**物料需求单**:配什么 / 配多少 / 怎么攒(哪批料服务哪几批,**确定性贪心打包**,非最优 lot-sizing)/ 原料需求 / 效期窗 / "1 次 vs N 次"经济建议。
- 人**审阅 / 修改**(拆攒批、改量、强制单独配、改用现有库存)。
- **确认 = 冻结物料计划**(写入共享已确认台账;**提交时做版本校验 / 乐观锁**)。
- "能不能攒"此闸定**必要条件**(效期窗 + 数量,靠锚点粗时间线);但**CIP 容量可行性要到闸 2 才算**(攒批动机正是省 CIP 这个真瓶颈)→ 闸1 的"可攒"是**必要非充分**。
- **闸1⇄闸2 回退协议(已定 C16,轻量 —— 物料单人工审核兜底,用户判"问题不大")**:① 闸1"可攒"标为"待闸2确认";② 闸2 塞不下 → 单向、有限次、带冲突证据退回闸1 拆该攒批;**多数由 §5 末招(主工艺 CIP 窗内微调)就地化解,残留交人工审核**。不搞重协议。

**闸 2 · 任务派发(时间的闸)**
- 输入:冻结的物料计划。
- 跑 ② 时序(STN)+ ③ 资源落点 + ④ 消解 → 排定工序。
- 人审 → **派发** → 喂排班(solver_v4/v5)+ 冲突 / 增援报告。

**两闸节奏不同**:料闸**只在物料计划真变时重开**(不为微调 CIP 一小时就重批攒料);时间闸更频繁。确认过的料计划若因某批拖延而**超效期 → 标 stale → 自动提示重审**(复用 plan/actual 粘性 + stale)。

**架构收益**:① 把"跨批并发盲区"从"持续并行"**降级成"确认提交点的版本校验"**(成熟可控);② 攒批经济账(1 次 vs N 次)以**提案**形式交人定夺;③ 料 / 时间不同前置期、不同负责人,分闸尊重现实;④ 与 **GMP 物料复核放行 → 生产指令派发** 同构,审计天然认;⑤ "最小数量账"活在料闸上、有人背书。
