# 补足工时日期优先级分析

## 一、日期筛选条件

### 必须满足的条件（硬约束）

一个日期必须同时满足以下条件才能被选为补足工时的候选日期：

1. **未被锁定**：`!isLocked`（不在`stats.lockedDates`中）
2. **是工作日**：`isWorkday`（从`calendar_workdays`表查询，`is_workday = 1`）
3. **未达每日上限**：`existingHours < maxDailyHours`（默认12小时）

```typescript
if (!isLocked && isWorkday && existingHours < maxDailyHours) {
  // 可以添加班次
}
```

### 排除的日期

以下日期**不会**被选为补足工时的候选日期：

- ❌ **锁定日期**：已锁定的排班日期（`stats.lockedDates`）
- ❌ **非工作日**：周末、法定节假日（`is_workday = 0`）
- ❌ **已达每日上限**：已有工时 ≥ 12小时

## 二、优先级计算规则

### 优先级分值

系统为每个可用日期计算优先级分值，分值越高，优先级越高：

| 日期类型 | 优先级分值 | 说明 |
|---------|-----------|------|
| **高峰日 + backup需求** | `200 + backupNeeded × 10` | 最高优先级，backup需求越多，优先级越高 |
| **高峰日（无backup）** | `100` | 较高优先级 |
| **工作日（无排班）** | `50` | 中等优先级 |
| **工作日（有排班但未达上限）** | `0` | 较低优先级 |

### 优先级计算代码

```typescript
let priority = 0;
if (isPeakDay && backupNeeded > 0) {
  priority = 200 + backupNeeded * 10; // backup需求越多，优先级越高
} else if (isPeakDay) {
  priority = 100;
} else if (existingHours === 0) {
  priority = 50;
}
// 否则 priority = 0
```

### 优先级示例

假设某季度有以下日期：

| 日期 | 类型 | backup需求 | 已有工时 | 优先级 |
|------|------|-----------|---------|--------|
| 2025-10-05 | 高峰日 | 3人 | 0h | **230** (200 + 3×10) |
| 2025-10-10 | 高峰日 | 2人 | 0h | **220** (200 + 2×10) |
| 2025-10-15 | 高峰日 | 0人 | 0h | **100** |
| 2025-10-20 | 工作日 | 0人 | 0h | **50** |
| 2025-10-25 | 工作日 | 0人 | 4h | **0** |

## 三、排序规则

### 主要排序规则

```typescript
availableDates.sort((a, b) => {
  if (a.priority !== b.priority) {
    return b.priority - a.priority; // 优先级高的在前
  }
  return a.existingHours - b.existingHours; // 工时少的在前
});
```

排序逻辑：
1. **第一优先级**：按优先级分值降序（分值高的在前）
2. **第二优先级**：优先级相同时，按已有工时升序（工时少的在前）

### 排序示例

假设有以下可用日期：

| 日期 | 优先级 | 已有工时 | 排序后位置 |
|------|--------|---------|-----------|
| 2025-10-05 | 230 | 0h | **1**（最高优先级） |
| 2025-10-10 | 220 | 0h | **2** |
| 2025-10-15 | 100 | 0h | **3** |
| 2025-10-20 | 50 | 0h | **4** |
| 2025-10-25 | 50 | 2h | **5**（优先级相同，但工时更多） |
| 2025-10-30 | 50 | 4h | **6** |

## 四、补足顺序

### 补足流程

1. **遍历所有可用日期**：按排序后的顺序依次处理
2. **计算可添加工时**：`min(剩余需要工时, 每日剩余容量)`
3. **添加班次**：直到满足所需工时或没有可用日期

```typescript
let remainingHours = neededHours;
for (const dateInfo of availableDates) {
  if (remainingHours <= 0) break;
  
  const maxAddableHours = Math.max(0, maxDailyHours - existingHours);
  const hoursToAdd = Math.min(remainingHours, maxAddableHours);
  
  // 添加班次...
  remainingHours -= hoursToAdd;
}
```

### 补足示例

假设员工需要补足40小时工时：

**可用日期（已排序）**：
1. 2025-10-05（高峰日，backup需求3人，优先级230，已有0h）
2. 2025-10-10（高峰日，backup需求2人，优先级220，已有0h）
3. 2025-10-15（高峰日，优先级100，已有0h）
4. 2025-10-20（工作日，优先级50，已有0h）
5. 2025-10-25（工作日，优先级50，已有2h）

**补足过程**：
1. **2025-10-05**：添加8小时（剩余32小时）
2. **2025-10-10**：添加8小时（剩余24小时）
3. **2025-10-15**：添加8小时（剩余16小时）
4. **2025-10-20**：添加8小时（剩余8小时）
5. **2025-10-25**：添加8小时（剩余0小时，完成）

## 五、特殊处理

### 1. 高峰日backup人员

如果日期是高峰日且有backup需求，会特殊处理：


### 2. 每日工时限制

- **默认上限**：12小时
- **计算方式**：`maxAddableHours = maxDailyHours - existingHours`
- **限制**：单次添加不超过每日剩余容量

### 3. 无可用日期处理

如果没有可用日期，会记录警告：


## 六、优先级策略总结

### 优先级从高到低

1. **高峰日 + backup需求**（优先级 ≥ 200）
   - 原因：高峰日需要更多人员支持，backup需求紧急
   - 策略：优先满足高峰日的backup需求

2. **高峰日（无backup）**（优先级 = 100）
   - 原因：高峰日操作任务多，需要更多人员
   - 策略：在高峰日补足工时，提高生产效率

3. **工作日（无排班）**（优先级 = 50）
   - 原因：无排班的日期更容易安排
   - 策略：优先在无排班的日期补足，避免单日工时过高

4. **工作日（有排班但未达上限）**（优先级 = 0）
   - 原因：已有排班，但未达每日上限
   - 策略：在已有排班的日期补充工时，但优先级较低

### 同优先级时的选择

当优先级相同时，选择**已有工时更少**的日期：
- 原因：均匀分布工时，避免单日工时过高
- 策略：优先在工时少的日期补足

## 七、实际应用场景

### 场景1：季度补足（需要补足100小时）

**季度范围**：2025-10-01 至 2025-12-31

**补足顺序**：
1. 优先选择高峰日（10月、11月、12月的高峰日）
2. 其次选择无排班的工作日
3. 最后选择已有排班但未达上限的工作日

**结果**：工时优先分配到高峰日，提高生产效率

### 场景2：月度补足（需要补足20小时）

**月度范围**：2025-10-01 至 2025-10-31

**补足顺序**：
1. 优先选择10月的高峰日（如果有）
2. 其次选择10月无排班的工作日
3. 最后选择10月已有排班但未达上限的工作日

**结果**：工时在月度内均匀分布

### 场景3：高峰日backup补足

**高峰日**：2025-10-05（backup需求3人）

**补足顺序**：
1. **最高优先级**：2025-10-05（优先级230）
2. 其他日期按正常优先级排序

**结果**：优先在高峰日补足backup人员，确保高峰日有足够人员支持

## 八、优化建议

### 当前策略的优点

1. ✅ **高峰日优先**：确保高峰日有足够人员
2. ✅ **backup需求优先**：满足紧急的backup需求
3. ✅ **均匀分布**：优先选择工时少的日期，避免单日工时过高

### 可能的改进

1. **考虑月度均匀分布**：
   - 当前策略可能导致工时集中在某些月份
   - 建议：在季度补足时，考虑月度均匀分布

2. **考虑连续工作天数**：
   - 当前策略可能产生连续工作过多天
   - 建议：优先选择距离上次工作日较远的日期

3. **考虑员工偏好**：
   - 当前策略未考虑员工偏好
   - 建议：在优先级计算中加入员工偏好因素
