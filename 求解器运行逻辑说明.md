## 什么是求解器

**求解器（Solver）** 是一个自动排班的"智能大脑"，它就像一位非常细心的排班经理：

- **输入**：告诉它有哪些工作任务、有哪些员工、每个员工的技能水平、以及各种规则要求
- **处理**：自动思考如何安排最合理，既要满足所有规则，又要尽可能高效
- **输出**：给出一份详细的排班表，告诉你哪个员工在什么时间做什么工作

## 第一阶段：数据准备

### 🎯 目标
收集所有需要的信息，就像排班前先把所有资料摆在桌面上。

### 📥 需要哪些数据？

#### 1. **工作任务信息（operationDemands）**
每个需要安排人员的工作任务，包括：
- 任务名称（例如：搅拌作业、监控操作）
- 计划开始时间和结束时间
- 需要多少人
- 需要什么技能等级

**举例**：
```
任务：搅拌作业-001
时间：2025-12-01 08:00 到 16:00
需要人数：2人
技能要求：搅拌技能 3级以上
```

#### 2. **员工信息（employeeProfiles）**
所有可用的员工资料，包括：
- 员工姓名和编号
- 岗位级别（一线员工、组长、经理等）
- 拥有的技能和等级
- 每月应工作的标准小时数

**举例**：
```
员工：张三（ID: 101）
岗位：一线员工
技能：搅拌技能 5级、监控技能 3级
标准工时：每月 176 小时
```

#### 3. **日历信息（calendar）**
哪些日子是工作日、休息日、节假日：
- 每一天的日期
- 是否工作日
- 是否三倍工资日（如春节）

**举例**：
```
2025-12-01：工作日
2025-12-07：休息日
2025-01-01：三倍工资节假日
```

#### 4. **锁定信息（lockedOperations / lockedShifts）**
已经确定不能更改的安排：
- 某些任务必须由特定员工执行
- 某些员工已经安排了固定班次

**举例**：
```
任务"质检-005"必须由李四执行（专项认证）
张三在 12月5日 已锁定夜班
```

#### 5. **配置参数（config）**
各种规则开关和阈值：
- 每月工时容差（例如：±16小时）
- 夜班后最少休息天数
- 最多连续工作天数
- 是否优先安排一线员工
- 是否启用车间公平性等

## 第二阶段：建立约束规则

### 🎯 目标
把所有排班规则转换成数学约束，就像给"智能大脑"设定思考的边界。

### 📏 核心约束规则

#### 1. **任务人员需求约束**
> **保证每个任务都有足够的合格人员**

- 检查每个任务需要几个人
- 找出所有符合技能要求的员工
- 确保至少有要求数量的员工被分配

**举例**：
```
搅拌作业-001 需要 2人，技能要求 3级
可选员工：张三(5级)、李四(3级)、王五(4级)
约束：从这3人中至少选择2人
```

#### 2. **时间冲突约束**
> **同一个员工不能同时做两件事**

- 检查所有任务的时间窗口
- 如果两个任务时间有重叠
- 则同一员工最多只能选其中一个

**举例**：
```
任务A：08:00-12:00
任务B：10:00-14:00
张三不能同时做A和B（时间重叠）
```

#### 3. **每日工作时长约束**
> **记录每个员工每天工作多少分钟**

- 统计员工当天所有任务的总时长
- 防止单日工作时间过长（硬性限制通常 ≤ 24小时）
- 配合月度工时约束使用

#### 4. **每月工时约束（可配置开关）**
> **保证每个员工的月度工作时间符合标准**

- 计算每月应有的标准工时（工作日数 × 8小时）
- 允许一定容差（例如 ±16小时）
- 三倍工资日可排除在外

**举例**：
```
某月有 22 个工作日
标准工时 = 22 × 8 = 176 小时
容差 ±16 小时
有效范围：160 ~ 192 小时
```

#### 5. **夜班休息约束（可配置开关）**
> **夜班后必须有足够的休息时间**

- 识别哪些任务是夜班（通常21:00后开始或跨夜11小时以上）
- 如果员工上了夜班
- 后续N天（可配置，默认2天）禁止安排工作

**举例**：
```
张三在 12月1日 上了夜班
则 12月2日、12月3日 自动休息
```

#### 6. **连续工作限制（可配置开关）**
> **防止员工连续工作太多天**

- 设定最大连续工作天数（例如：6天）
- 在任意连续N+1天窗口内（例如7天）
- 最多只能工作N天（例如6天）

**举例**：
```
最大连续工作 6 天
在任意连续 7 天内，最多工作 6 天
保证至少有 1 天休息
```

#### 7. **季度工时约束（可配置开关）**
> **确保每季度至少达到标准工时**

- 只在日历数据覆盖完整季度时启用
- 计算季度标准工时
- 不包括三倍工资日

#### 8. **车间公平性约束（可配置开关）**
> **让同岗位员工的工作量尽可能均衡**

- 将员工按岗位级别分组（一线、组长、经理等）
- 计算同组员工的月度工时平均值
- 限制个人工时与平均值的偏差在容差范围内

**举例**：
```
一线员工组有 5 人
本月平均工时应为 176 小时
容差 ±2 小时
每人工时应在 174 ~ 178 小时之间
```

#### 9. **锁定约束**
> **强制执行已确定的安排**

- 如果任务已锁定给特定员工
- 则必须分配该员工，不能更改

#### 10. **共享组约束**
> **同一批次的相关任务使用相同的人员团队**

- 识别需要共享人员的任务组（例如同一批次的多个阶段）
- 选定锚定任务（通常是需要人最多的）
- 其他任务的人员必须从锚定任务的人员中选择

## 第三阶段：优化求解

### 🎯 目标
在满足所有约束的前提下，找到最优的排班方案。

### 🧮 优化目标（从高到低优先级）

#### 1. **最小化出勤天数（基础目标）**
- 尽可能减少员工的出勤天数
- 这样可以集中工作，增加休息日

#### 2. **优先使用一线员工（可选）**
- 如果配置启用
- 优先安排一线员工（FRONTLINE）
- 对管理人员（组长、经理）的使用增加惩罚权重

#### 3. **减少三倍工资日出勤（可选）**
- 如果配置启用
- 对三倍工资日（如春节）的出勤增加惩罚
- 减少节假日用工成本

#### 4. **车间公平性（可选）**
- 如果启用车间公平性
- 对工时不均衡增加惩罚
- 促使同岗位员工工作量趋于一致

### ⚙️ 求解引擎

使用 **Google OR-Tools CP-SAT Solver**（约束规划求解器）：
- 这是一个强大的优化引擎
- 可以处理复杂的约束和大规模问题
- 会在限定时间内（默认30秒，可配置）寻找最优解

### 🔄 求解过程

1. **构建数学模型**：把所有约束翻译成数学不等式
2. **设定优化目标**：告诉求解器什么样的方案更好
3. **启动计算**：求解器开始智能搜索
4. **监控状态**：
   - 如果找到最优解（OPTIMAL）→ 完美！
   - 如果找到可行解（FEASIBLE）→ 可接受
   - 如果无解（INFEASIBLE）→ 说明约束条件冲突，无法满足所有要求

## 第四阶段：结果生成

### 🎯 目标
将求解器的数学结果转换成人类可读的排班表。

### 📊 生成内容

#### 1. **任务分配表（assignments）**
明确每个任务由哪些员工执行：

```
任务ID    | 员工ID  | 员工姓名
----------|---------|----------
1001      | 101     | 张三
1001      | 102     | 李四
1002      | 103     | 王五
```

#### 2. **班次计划表（shiftPlans）**
为每个员工生成每日的班次信息：

```
日期       | 员工   | 班次类型   | 工作时长 | 开始时间 | 结束时间
-----------|--------|-----------|----------|----------|----------
2025-12-01 | 张三   | 生产班    | 8小时    | 08:00   | 16:00
2025-12-01 | 李四   | 生产班    | 8小时    | 08:00   | 16:00
2025-12-02 | 张三   | 休息      | 0小时    | -       | -
```

**班次类型说明**：
- **PRODUCTION（生产班）**：有具体任务的工作日
- **BASE（基础班）**：工作日但未分配具体任务
- **REST（休息）**：休息日

#### 3. **班次匹配逻辑**
系统会智能匹配标准班次：
- 根据员工当天的最早开始时间和最晚结束时间
- 与预定义的班次（早班、中班、晚班、夜班等）进行匹配
- 如果时间特殊则标记为"临时班次（TEMP）"

**举例**：
```
员工任务时间：08:00-16:00
匹配班次：早班（08:00-17:00）
容差：30分钟
```

#### 4. **跳过的任务（skippedOperations）**
记录无法安排的任务及原因：
- 没有符合技能要求的员工
- 锁定的员工不存在
- 约束条件无法满足

## 第五阶段：数据存储

### 🎯 目标
将排班结果保存到数据库，供后续查询和展示。

### 💾 存储的数据表

#### 1. **batch_personnel_assignments（批次人员分配表）**
记录每个操作任务的人员分配：

| 字段名称           | 说明                   | 示例值              |
|--------------------|------------------------|---------------------|
| operation_plan_id  | 操作计划ID             | 1001                |
| employee_id        | 员工ID                 | 101                 |
| assignment_date    | 分配日期               | 2025-12-01          |
| shift_code         | 班次代码               | EARLY               |

#### 2. **employee_shift_plans（员工班次计划表）**
记录每个员工每天的班次计划：

| 字段名称       | 说明           | 示例值              |
|----------------|----------------|---------------------|
| employee_id    | 员工ID         | 101                 |
| date           | 日期           | 2025-12-01          |
| plan_type      | 计划类型       | PRODUCTION          |
| plan_hours     | 计划工时       | 8.0                 |
| shift_code     | 班次代码       | EARLY               |
| shift_name     | 班次名称       | 早班                |
| start_time     | 开始时间       | 2025-12-01T08:00:00 |
| end_time       | 结束时间       | 2025-12-01T16:00:00 |

### 🔄 存储流程

1. **清理旧数据**（如果是重新排班）
   - 删除指定时间范围内的旧排班记录
   
2. **批量插入新数据**
   - 将求解器生成的任务分配记录批量写入
   - 将班次计划记录批量写入
   
3. **事务保护**
   - 使用数据库事务确保数据一致性
   - 如果存储过程出错，自动回滚，不会留下不完整的数据

4. **返回结果**
   - 向前端返回成功消息
   - 包含分配数量、跳过任务等统计信息

## 🎬 完整案例演示

### 场景设定
某制药车间需要安排2025年12月份的排班，有以下情况：

**工作任务**：
- 发酵罐监控（每天2人，需要监控技能3级）
- 搅拌作业（每天1人，需要搅拌技能4级）

**员工**：
- 张三：一线员工，监控5级、搅拌5级
- 李四：一线员工，监控3级、搅拌3级
- 王五：组长，监控4级、搅拌6级

**规则要求**：
- 每月标准工时 176小时（22个工作日 × 8小时）
- 容差 ±16小时
- 夜班后至少休息2天
- 最多连续工作6天
- 优先使用一线员工

### 执行过程

#### 第一阶段：数据准备 ✅
- 读取12月份所有工作任务
- 读取3名员工的信息和技能
- 读取12月份日历（22个工作日、8个休息日）
- 读取配置参数

#### 第二阶段：建立约束 ✅
- 为每个任务创建人员分配变量
- 添加技能匹配约束（李四刚好3级，可以做监控但不能做4级以上搅拌）
- 添加时间冲突约束
- 添加月度工时约束（160-192小时）
- 添加夜班休息约束
- 添加连续工作约束（最多6天）

#### 第三阶段：优化求解 ✅
- 启动CP-SAT求解器
- 优化目标：最小化出勤天数 + 优先一线员工
- 求解器找到最优解
- 结果：
  - 张三：180小时，20天出勤
  - 李四：176小时，18天出勤
  - 王五：32小时，4天出勤（仅在必要时补充）

#### 第四阶段：结果生成 ✅
- 生成 `batch_personnel_assignments` 记录：660条（22天 × 30个任务）
- 生成 `employee_shift_plans` 记录：90条（3人 × 30天）

#### 第五阶段：数据存储 ✅
- 清理12月份旧排班数据
- 批量插入新分配记录
- 批量插入新班次计划
- 提交事务
- 返回成功消息

## 📌 常见问题

### Q1: 如果求解器返回"无解"怎么办？
**A**: 说明约束条件太严格，无法同时满足所有要求。可以：
- 放宽工时容差
- 减少锁定任务
- 增加可用员工
- 关闭部分非必要约束（如车间公平性）

### Q2: 求解时间可以延长吗？
**A**: 可以，通过调整 `solverTimeLimit` 参数。但更长时间并不一定得到更好的解，通常30-60秒已足够。

### Q3: 为什么有些员工工作特别多？
**A**: 可能原因：
- 该员工技能特殊，别人无法替代
- 未启用车间公平性约束
- 锁定了太多任务给该员工

### Q4: 如何确保关键任务不被跳过？
**A**: 
- 确保有足够的合格员工
- 避免在同一时间锁定太多员工
- 检查技能等级要求是否过高

## 🎯 总结

求解器的运行逻辑可以概括为：

1. **收集信息** → 任务、员工、日历、规则
2. **设定规则** → 转换为数学约束
3. **智能计算** → 使用优化引擎寻找最佳方案
4. **生成结果** → 任务分配表 + 班次计划表
