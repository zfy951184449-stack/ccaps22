# 夜班休息约束升级说明

## 更新内容

将夜班休息约束从**纯软约束**升级为**混合约束模式**，兼顾劳动保护和调度灵活性。

## 新的约束模式

### 🔴 硬约束：夜班后第1天必须休息

- **不可违反**：夜班后的第二天（offset=1）**必须**安排为REST
- **适用范围**：所有类型的班次（PRODUCTION和BASE都不允许）
- **逻辑**：
  ```python
  # 如果上夜班，第二天不能上班（必须休息）
  model.Add(p == 0).OnlyEnforceIf(night_flag)
  model.Add(b == 0).OnlyEnforceIf(night_flag)
  ```

### 🟡 软约束：夜班后第2天尽可能休息

- **可以违反**：如果人手紧张，可以在夜班后第3天（offset=2）安排工作
- **有惩罚**：每次违反会在目标函数中增加 **100** 分的惩罚
- **控制参数**：`nightShiftPreferredRestDays`（默认2天）
- **逻辑**：
  ```python
  # 如果上夜班且第二天上班，则违反软约束
  viol = model.NewBoolVar(...)
  night_rest_penalty_vars.append(viol)
  model.Add(p2 == 0).OnlyEnforceIf([night_flag, viol.Not()])
  ```

## 配置参数

### enforceNightRest (布尔值，默认: true)
- 是否启用夜班休息约束
- 如果设为 `false`，所有夜班休息约束都不生效

### nightShiftPreferredRestDays (整数，默认: 2)
- 夜班后的优选休息天数
- 用于软约束（第2天）
- 如果设为小于2，则不会应用第2天的软约束

### ~~nightShiftMinimumRestDays~~ (已弃用)
- 此参数已不再使用
- 第1天的休息现在是硬编码的硬约束（不可配置）
- 保留此参数仅为向后兼容

## 示例场景

### 场景1：正常情况
```
Day 1: NIGHT_SHIFT (夜班)
Day 2: REST (硬约束，必须休息)
Day 3: REST (软约束，优选休息)
Day 4: BASE/PRODUCTION (可以正常排班)
```

### 场景2：人手紧张
```
Day 1: NIGHT_SHIFT (夜班)
Day 2: REST (硬约束，必须休息)
Day 3: PRODUCTION (违反软约束，但允许，惩罚+100分)
Day 4: BASE/PRODUCTION (正常)
```

### 场景3：不允许的情况（求解失败）
```
Day 1: NIGHT_SHIFT (夜班)
Day 2: PRODUCTION (❌ 违反硬约束，求解器会拒绝此解）
```

## 优势

### ✅ 劳动保护
- 确保员工夜班后至少休息1天（符合劳动法基本要求）
- 降低员工过度疲劳的风险

### ✅ 调度灵活性
- 在紧急情况下，可以在第3天安排工作
- 避免因过于严格的约束导致无解

### ✅ 优化目标
- 求解器会尽量安排2天休息
- 只在必要时才违反软约束

## 罚分权重

当前夜班休息软约束的罚分为 **100分/次**，在目标函数中的优先级：

```python
# 目标函数组成（从高到低）：
1. 操作分配缺口 (slack_var)       - 最高优先级
2. 管理层使用 (manager_vars)      - 100分/人
3. 夜班休息违反 (night_rest)      - 100分/次  ← 当前
4. 连续工作违反 (consecutive)     - 50分/次
5. 月度工时违反 (monthly_hours)   - 5分/小时
```

夜班休息的权重与管理层使用相同，表示这是一个重要但不是最高的优化目标。

## 验证方法

1. **创建包含夜班操作的批次**
2. **运行自动排班**
3. **检查结果**：
   - 夜班后第1天应该全是REST
   - 夜班后第2天尽可能是REST（但可以有例外）
   - 如果第2天不是REST，求解器应该显示相应的惩罚分数

## 回滚方法

如果需要恢复到纯软约束模式，可以：

1. 将硬约束部分改回软约束
2. 或者通过配置设置 `enforceNightRest: false` 完全禁用
