# Codex Plan Collaboration Rules

目标：在关键前提不清楚时，优先澄清高价值问题，并把复杂工作的思考沉淀到仓库里，而不是依赖聊天上下文。

## 1. 何时优先提问

1. 用户目标有两种及以上合理解释，且会导向不同实现路径。
2. 缺少关键验收标准，继续推进大概率会做出“技术上能跑但不是用户要的东西”。
3. 涉及高风险动作而边界不清楚：
   - 删除、覆盖、批量迁移、持久化结构调整
   - API 契约变更
   - 排程/生物制药硬约束语义变更
4. 本地代码、文档、运行现状与用户描述明显冲突，无法判断哪个是 source of truth。
5. 有多个可行方案，但取舍依赖用户偏好而不是纯技术判断。

## 2. 提问方式

1. 问题数量控制在 `1-3` 个，只问会改变后续方案的关键问题。
2. 问题要短、直接、可决策，避免泛泛地让用户“再详细说说”。
3. 每轮提问都应有清晰目的：
   - 确认范围
   - 确认 source of truth
   - 确认取舍偏好
4. 如果只有一个关键缺口，就只问一个问题。

## 3. 问完之后怎么做

1. 收到答复后，先用 `1-2` 句话复述更新后的理解，再继续执行。
2. 如果用户暂时没有回复，但仓库中存在明确 source of truth，可以显式引用依据并在低风险范围内继续推进。
3. 如果仍然存在高风险不确定性，继续追问；不要为了推进速度擅自补齐关键业务前提。
4. 对于跨会话、多步骤、需要决策日志的工作，把计划写入 `docs/exec-plans/active/`。

## 4. 避免过度提问

1. 能通过代码、规则、测试、迁移、历史实现确认的问题，先自己查。
2. 对低风险实现细节，不要频繁打断用户。
3. 只有当“不问会显著增加返工概率”时，才升级为提问。

## 5. 输出要求

1. 明确告诉用户你不确定的点是什么。
2. 让用户看得出：答案会影响哪个实现方向或验收结果。
3. 在得到回答后，把方案收敛成可执行步骤，而不是停留在持续讨论状态。
