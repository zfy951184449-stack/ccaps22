架构问题 (Architectural)

God Method 反模式
solver.py:solve()
High
问题描述

solve() 方法长达 ~470 行，包含变量创建、约束应用、目标函数构建、求解执行、结果提取等所有逻辑，严重违反单一职责原则 (SRP)。

影响

• 难以单独测试各个阶段
• 代码复用性差
• 新增功能时容易引入 Bug
• 多人协作时容易冲突

修复方案

重构为多个私有方法：

class SolverV4: def solve(self, req: SolverRequest): # 1. Build variables assignments, vacancy_vars, shift_assignments = self._build_variables(req) # 2. Apply constraints self._apply_constraints(assignments, shift_assignments, req) # 3. Build objective self._build_objective(assignments, shift_assignments, vacancy_vars, req) # 4. Run solver status = self._run_solver(callback) # 5. Extract solution return self._extract_solution(...)

岗位分配约束（"每个岗位必须分配1人"）直接写在 solve() 方法中，而非封装为独立的 ConstraintModule。这与已有的 OnePositionConstraint 功能重复。

变量规模不可控
solver.py:164-169
High
问题描述

班次变量 shift_assignments 的创建采用全笛卡尔积：Days × Employees × Shifts。即使 90% 的组合不合法（如员工A不可能上夜班），也会全部创建变量。

影响

• 变量数量级：14天 × 50人 × 3班次 = 2100 变量
• 大量无效变量拖慢求解速度
• 内存占用增加
• 违反"稀疏结构"原则

修复方案
# Before (Full Cartesian Product) for date in dates: for emp_id in all_employees: for shift in req.shift_definitions: shift_assignments[(emp_id, date, shift.shift_id)] = ... # After (Sparse Creation) + for (emp_id, date, shift_id) in self._get_valid_shift_combinations(req): + shift_assignments[(emp_id, date, shift_id)] = ...
缺乏 Dead Variable Pruning
solver.py (全局)
Medium
问题描述

对于无任何相关 Operation 覆盖的 shift_assignments 变量（例如某天没有任何工序安排），仍然会被创建，成为"死变量"。

影响

• 增加求解器的搜索空间
• 降低求解效率

修复方案

在变量创建后，分析 Operation 覆盖范围，删除或不创建"死变量"。

健壮性问题 (Robustness)

线程安全隐患
solver.py:351-387 / callback.py
High
问题描述

monitor_loop 线程与主 Solver 线程共享 callback 状态对象，但 should_stop、best_objective、solution_count 等字段无锁保护。

影响

• 在高并发场景下可能出现数据竞争
• 极端情况下导致状态读取不一致
• 难以复现的偶发性 Bug

修复方案
import threading class APICallback: def __init__(self, ...): + self._lock = threading.Lock() self.should_stop = False def request_stop(self, reason): + with self._lock: self.should_stop = True
Callback 依赖同步 HTTP
callback.py:push_progress()
Medium
问题描述

push_progress() 使用同步 requests.post() 调用，若 Backend 响应慢或网络超时，会阻塞 Callback，进而影响求解性能。

影响

• 网络抖动时求解器被"拖慢"
• 前端进度更新不及时

修复方案

使用异步队列 + 后台线程推送，或使用 requests 的 timeout 参数限制阻塞时间。
