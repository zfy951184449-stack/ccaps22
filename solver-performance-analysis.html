🔍 根因分析（按影响排序）
根因 #1 · 最高优先级
候选人预过滤形同虚设
DataAssemblerV4 有预过滤逻辑，但实测中位数 27/31 名员工成为候选人——过滤率仅 ~13%。

根本原因：过滤条件仅检查资质（qualification_id + level），但生物制药岗位资质设置宽松，大多数员工都具备，导致候选池基本未缩减。

影响：Assignment 变量从理想的 ~6,000 膨胀到 21,905，搜索空间扩大约 3~4 倍，CP-SAT 需要更长时间传播约束。
影响：搜索空间 ×3~4，是最主要慢的原因
根因 #2 · 最高优先级
443 个操作超出单次求解合理规模
一个月内 443 个 operation_plan_id，对应 31 名员工。

UniqueEmployee 约束对每名员工做操作对时间重叠检测：理论最大对数 = 443×442/2 ≈ 98,000 对/员工 × 31 = 3,034,993 次对比。即使 sweep-line 有剪枝，仍是模型构建主要瓶颈。

84 种 operation_id 重复出现 443 次，说明同一工序在多批次中重复——适合按批次或按部门分窗口求解。
影响：约束构建 O(E×N²)，443 ops 时已超阈值
根因 #3 · 高优先级
MinimizeDeviation 目标存在 O(E²) 内层循环
objectives/minimize_deviation.py 的实现：
for emp_id in all_employees: # 31次 for (e_id,date,shift_id),var # 4650次 in shift_assignments.items(): # ← 全量扫描！ if e_id == emp_id: ... 实际 144,150 次迭代。正确做法是预先按员工分组，降至 4,650 次。
同样问题存在于 balance_night_shifts.py。
影响：模型构建时间 × 31，可优化为 1x
根因 #4 · 高优先级
log_search_progress = True 在生产环境开启
solver.parameters.log_search_progress = True 硬编码在 SolverV4.__init__。

CP-SAT 使用 8 个 worker 并行搜索，每个 worker 在找到更优解时都会写一行日志。对于 300s 求解，可能产生数百条日志，每次日志调用都是系统调用级别的 I/O，在求解过程中持续竞争 GIL，并干扰 CP-SAT 的搜索节奏。

应改为仅在 SOLVER_DEBUG=1 时开启。
影响：求解中持续 I/O，预计拖慢 5~15%
根因 #5 · 高优先级
strptime 在内层循环中调用
work_days_limit.py、night_rest.py 中在员工×日期的内层循环里调用 datetime.strptime(d, "%Y-%m-%d")。

实测 strptime 比 date.fromisoformat() 慢 9.1 倍（0.221s vs 0.024s per 100k 次）。

简单替换即可消除此开销，且 date.fromisoformat() 从 Python 3.7+ 均可用。
影响：日期解析开销减少 9x（10ms→1ms）
根因 #6 · 高优先级
Python sum() 替代 LinearExpr.Sum()
多处约束使用 sum(coef * var for ...)（Python 内置 sum）构建 CP-SAT 表达式，而非 cp_model.LinearExpr.WeightedSum(vars, coeffs)。

Python sum 对 CP-SAT 对象会在 Python 层反复创建临时 Add 节点，而 LinearExpr.WeightedSum 直接在 C++ 层批量构建，减少 Python→C++ 边界调用次数。

影响出现在 standard_hours.py（每员工每月构建一次）和目标函数中。
影响：模型构建时间减少 20~40%
根因 #7 · 中优先级
无 Warm Start（每次重建模型）
每次 /api/v4/solve 调用都从空白模型开始：SolverV4() → cp_model.CpModel()。

当用户只改了 1 个锁定操作后重新求解，仍需跑满 300s。

CP-SAT 支持 model.AddHint(var, val) 注入上次解作为初始提示。加入暖启动后，轻微改动的重算可在 10~30s 内完成。
影响：重算场景从 300s → 10~30s
根因 #8 · 中优先级
num_workers=8 硬编码，不适配容器
CP-SAT 8 路 worker 在本地 MacBook 可能合适，但在容器（2 vCPU）或 CI 环境中，8 workers 会产生严重线程竞争，反而比 2~4 workers 更慢。

应改为：min(8, os.cpu_count() or 4)，或通过 SOLVER_NUM_WORKERS 环境变量配置。
影响：容器部署时可能比优化前更慢
🛠 具体修复方案（代码级）
以下为最高优先级的代码级修改，每个修复独立可实施
Fix #1 · MinimizeDeviation: O(E²) → O(1) 预分组
❌ 当前实现（O(E × total_shift_vars)）
# objectives/minimize_deviation.py  第 89-93 行
for emp_id in all_employees:       # 31次外层
    emp_terms = []
    for (e_id, date, shift_id), var \
        in shift_assignments.items():  # 4650次内层
        if e_id == emp_id:             # 逐一比较
            ...
✅ 修复后（O(shift_vars) 预分组）
# 在构建 shift_assignments 后做一次预分组
from collections import defaultdict
by_emp = defaultdict(list)
for (emp_id, date, shift_id), var \
    in shift_assignments.items():
    by_emp[emp_id].append((shift_id, var))  # O(1)

# 目标函数中直接用预分组
for emp_id in all_employees:
    for shift_id, var in by_emp[emp_id]:   # 无内层全扫
        ...
Fix #2 · strptime → fromisoformat（9.1x 提速）
❌ 当前（work_days_limit.py, night_rest.py）
for d in dates:
    date_objs.append(
        datetime.strptime(d, "%Y-%m-%d").date()
    )
# 每次调用约 2.2μs，100员工×30天 = 6.6ms
✅ 修复后（Python 3.7+ 内置）
from datetime import date
for d in dates:
    date_objs.append(
        date.fromisoformat(d)
    )
# 每次约 0.24μs，快 9.1x
Fix #3 · LinearExpr.WeightedSum 替代 Python sum()
❌ 当前（standard_hours.py 等）
# Python sum 逐项构建 Add 节点
total_hours_expr = sum(
    coef * var
    for coef, var in emp_shift_terms
)
model.Add(total_hours_expr >= min_hours_scaled)
✅ 修复后（C++ 层批量构建）
vars_list  = [var  for _, var  in emp_shift_terms]
coefs_list = [coef for coef, _ in emp_shift_terms]
total_hours_expr = cp_model.LinearExpr.WeightedSum(
    vars_list, coefs_list
)
model.Add(total_hours_expr >= min_hours_scaled)
Fix #4 · 生产环境关闭搜索日志
❌ 当前（core/solver.py 第 36 行）
self.solver.parameters.log_search_progress = True
self.solver.parameters.num_workers = 8
self.solver.parameters.linearization_level = 2
✅ 修复后（按环境变量控制）
import os
debug = os.getenv("SOLVER_DEBUG","0") == "1"
self.solver.parameters.log_search_progress = debug
workers = int(os.getenv("SOLVER_WORKERS",
    str(min(8, os.cpu_count() or 4))))
self.solver.parameters.num_workers = workers
Fix #5 · 添加暖启动（Warm Start）
❌ 当前（每次从零开始）
# app.py: 每个请求新建求解器，无初始解
solver = SolverV4()
result = solver.solve(req)
# 用户改动 1 个锁定操作后重算：
# 仍需全量 300s
✅ 修复后（注入上次解作为 hint）
# 在 _build_variables 后，_run_solver 前：
if prev_solution := cache.get(req.window):
    for (op,pos,emp), var in assignments.items():
        hint_val = prev_solution.get((op,pos,emp),0)
        self.model.AddHint(var, hint_val)
# 轻微变更重算：10~30s 而非 300s
架构级优化 · 按批次分解求解窗口（最大效益）
一次性求解 443 个操作是当前慢的根本原因。443 个操作含 84 种操作类型，说明是多批次重叠。

建议：在后端 DataAssemblerV4 层按批次 ID（batch_id）或部门/Phase（USP/DSP）将操作分组，每组独立调用一次 /api/v4/solve：

批次 A（USP，~50 ops）→ Solve → 结果锁定
批次 B（DSP，~60 ops）→ Solve（含 A 的 frozen_assignments）→ 结果锁定
...
而非：所有批次 443 ops → 单次 Solve（当前）
每次求解变量从 21,905 降至 ~2,000~3,000，搜索空间从 2^21905 降至 2^3000 量级，预计求解时间从 300s 降至 10~30s/批次。已有 frozen_assignments 字段支持跨批次锁定。
