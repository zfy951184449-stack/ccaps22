🧮 求解器深度评估：OR-Tools CP-SAT 是否合适？

✅ 算法选型：CP-SAT 正确，不需要换
人员排班（Employee Scheduling / NSP）是经典 NP-hard 问题。CP-SAT 正是为此类问题设计的：
· 硬约束 + 软目标同时建模，天然支持可行性与最优性权衡
· 18 个约束模块解耦良好，每个模块独立可测
· 8 路并行 Worker 利用多核（Google 内部也用 CP-SAT 做排班）
· DataAssemblerV4 的候选人预过滤显著压缩搜索空间
· 相比 Gurobi/CPLEX：免费、无 License 风险，对此问题规模性能接近
🔴 部署方式：Flask WSGI 严重不适合
问题不在 CP-SAT，在 Flask 同步阻塞 + 单进程：
· /api/v4/solve 是 CPU 密集型，默认超时 300s（5分钟），期间整个服务无法响应
· 第二个求解请求进来时直接阻塞，无任务队列
· ACTIVE_CALLBACKS 是全局 dict，多进程部署时状态不共享，abort 失效
· 每次 solve 将完整请求 JSON dump 到 logs/ 磁盘，无清理策略
📐 问题类型与规模评估
问题本质
多资质约束的人员-操作分配问题（扩展 NSP），典型变量规模：
O(操作数 × 岗位数 × 候选员工数)
预过滤后每操作约 5~30 个候选人，布尔变量总量可控（估计 5k~50k），CP-SAT 处理此规模绰绰有余
硬约束复杂度
· 唯一分配（one_position）
· 班次覆盖（shift_assignment）
· 连续工作日上限（work_days_limit）
· 夜班后休息（night_rest）
· 共享组（share_group）
· 冻结区间（frozen_range）
← 这些用 CP-SAT 表达最自然
规划时域
支持子区间（solve_range）重算
但每次仍全量重建模型
无暖启动（solution hint）
→ 小改动触发全量 5 分钟求解
→ 是最大的用户体验痛点
⚠️ 最大功能缺口：资源约束接收但未实现
SolverRequest 已接收（Python Contract）
✓ resources（设备/房间列表）
✓ resource_calendars（设备可用时间）
✓ operation_resource_requirements（操作需要哪些设备）
✓ maintenance_windows（设备维护窗口）
constraints/ 目录中 无任何资源约束模块
✗ 设备冲突（同一设备同时只能一个操作使用）未建模

无暖启动 (Warm Start) 警告 每次小改动（锁定1个操作）仍触发全量 5 分钟重算 缓存上次解，用 model.AddHint(var, val) 注入为初始解提示，大幅加速收敛
ACTIVE_CALLBACKS 全局状态 警告 多进程部署时 abort 请求可能打到不同进程导致失效 迁移到 Redis 存储 callback 状态，abort 通过 Redis pub/sub 广播
无问题分解策略 建议 批次数/员工数增长时单一模型变量爆炸 按部门/Phase（USP→DSP）分解求解，或滚动时域（Rolling Horizon）
请求 JSON 全量落盘 建议 每次 solve 都把完整请求 JSON 写到 logs/，无清理 仅在 DEBUG 模式落盘，或保留最近 N 条，避免磁盘堆满
版本号 4.0.0-alpha 建议 alpha 版作为主求解链路，稳定性预期不明确 明确定义 alpha→beta→stable 质量门禁，通过后升版本号
